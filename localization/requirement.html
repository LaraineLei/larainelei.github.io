# 需求管理

需求

由需求方（一般为项目组）提出的翻译需求。

需求管理中，不同角色的操作权限：

操作/角色需求方项目负责人译员
创建✔️//
修改✔️//
查看✔️✔️/
提交✔️✔️/
审核/✔️/
验收✔️//

# 创建需求

# 创建流程

  • 登录本地化平台，在角色选择面板，选择角色为需求方
  • 或在导航栏右侧，切换角色为需求方
  • 在导航栏左侧，选择一个项目
  • 点击创建需求

  • 填写需求详情

    • 需求名称：用于简短描述需求特征。
    • 需求编码：用户唯一标志需求，用户可自定义。
    • 原语言：文件中已有的语言，目前只能选择一种。
    • 目标语言：需要译员进行翻译的语言，可多选种。
    • 期望完成时间：希望得到翻译文件的时间节点。
    • 需求类型：
      • 游戏类：需要按模板上传Excel文件。
      • 新闻稿类：上传Doc文件。
      • 游戏类（非模板）：支持多种格式的文件。
    • 附件文件：辅助翻译的资料，支持支持多种格式的文件。

  • 点击创建，需求创建成功

注意

需求创建成功后还在本地，可以继续修改，提交需求后，才能被项目负责人审核

# 需求模板

当需求类型为游戏类时，需要按照需求模板上传原文文件。模板中包含语言列和非语言列。

  • 语言列：需求的原语言、目标语言所对应的列。语言列必须要以语言代码开头，且按照语言代码:描述的格式进行填写，注意":"为英文格式,如en:英语。

  • 非语言列：备注列。用户可自定义列名，新增或删除非语言列。非语言列不能以语言代码开头,如模块。

注意

当原文文件中有一种以上的原语言时，由于需求的原语言只能选择一种，所有只能将需求原语言设为语言列，其他原语言设为为非语言列。

例如：

  • 创建游戏类需求时，将原语言设为简体中文，目标语言设为德语
  • 当原文文件有简体中文、英语两种语言时，简体中文、英语不能同时为语言列
  • 由于需求的原语言设为了简体中文，所以要将简体中文设为语言列，表头为cn:简体中文，将英语设为非语言列，表头为英语。

# 修改需求

当需求的翻译文件还没有提交时，需求方可以修改需求，但只能修改：

  • 目标语言
  • 原文件
  • 附加文件

需求修改后，可在需求详情-需求历史中查看修改记录。

# 需求详情

在任何状态下，都能查看需求详情

  • 点击查看详情按钮
  • 需求详情：需求的详细信息包括提出需求的需求方，负责需求的负责人，需求方创建需求时的填入的需求信息。可以下载原文文件、附加文件。
  • 需求历史：需求的操作历史。按照操作时间倒序排列。
  • 需求问题列表：需求方验收需求时，如果不通过，可创建多个问题。这些问题展示在问题列表中，也可前往问题管理查看详情。

# 提交需求

# 需求方提交需求

  • 可点击待提交状态的，查看所有待提交的需求，然后点击提交按钮进行提交
  • 也可在全部状态的需求中，点击点击提交按钮进行提交
  • 需求提交后，状态变为待审核，由项目负责人进行需求的审核

# 负责人提交需求

  • 可点击待提交状态的，查看所有待提交的需求，然后点击提交按钮进行提交
  • 也可在全部状态的需求中，点击点击提交按钮进行提交
  • 需求提交后，状态变为待验收，由需求方进行需求的验收

# 审核需求

# 接受需求

接受需求的同时，需要将需求拆分为以语言为单位的任务。

  • 预计开始时间：开始翻译需求的时间。
  • 预计完成时间：提交需求给需求方验收的时间。

接受需求后，需求将会在展示在部门排期日历中的预计开始时间-预计完成时间这个时间段。

  • 是否有预翻译：预翻译为有预先翻译的语言。预翻译语言的任务未完成时，其他语言的任务无法进行；预翻译语言的任务完成后，其他语言的任务可以看见预翻译语言的译文。

  • 本地化翻译：翻译途径为本地化部门内部译员进行翻译。需要选择一位可以进行原语言-目标语言双语翻译的译员。

  • 外包翻译：翻译途径为外包商翻译，本地化部门内部译员进行校对。需要选择一家外包商，和一位可以进行原语言-目标语言双语翻译的译员。

# 拒绝需求

  • 拒绝需求时，需要给出拒绝原因。
  • 需求被拒绝后，译员不会收到属于该需求的任务。

# 验收需求

# 通过需求

  • 点击通过按钮，将需求状态变为已通过。

# 不通过需求

不通过需求，需要通过创建问题来说明不通过的原因。

  • 项目负责人提交需求后，可以点击不通过按钮创建问题。
  • 一次只能创建关联一个目标语言的问题。
  • 一个需求中可以创建多个问题。
